## Verdict

**DEPRECATE as production tool.**

Remove `physics-analysis` from the mcp-ai-agent-guidelines routing manifest and the public skill surface. Keep the source code and tests as research scaffolding for future work.

## Trial Result

From Track C.2 trials across three real session contexts (code-review, bug-triage, architecture-planning):

| Category | Count |
|----------|-------|
| Load-bearing mappings | **0** |
| Decorative mappings | 2 |
| Misleading mappings | 1 |
| Correct rejections | 3 |

**Total trials:** 3 contexts, 6 attempted physics mappings (QM + GR per context).

**Load-bearing definition (from C.2):** A mapping that surfaces evidence *not already captured* by the existing workflow. A decorative mapping relabels an existing practice without adding falsifiable predictions or new evidence. A misleading mapping triggers on a pattern match but implies analytical rigor (e.g., Hawking-entropy calculations) that the underlying session does not support.

## Decision Rule Application

From the plan (verbatim):

> - ≥ 2 load-bearing across the 3 trials → **narrow**: keep the adapter, restrict to matched concerns, gate everything else.
> - 1 load-bearing → **research further**: design 1–2 more trials in that shape.
> - **0 load-bearing → DEPRECATE as production tool**: keep the code as research scaffolding, remove from the routing surface, mark with deprecation.

**We have 0 load-bearing.** The decision rule mandates **DEPRECATE**.

## The Corpus Confound

C.2 flagged a structural limitation: the session corpus (`session-*.json` files) contains only workflow-phase labels (`QUALITY`, `SECURITY`, `REPRODUCE`, `LOCATE`, `MEASURE`, `ARCHITECTURE`, etc.) and advisory-mode metadata. **The corpus does not include domain-specific engineering text.**

The physics adapter's 12 PhysicsConcern keywords (`flak`, `coupling`, `debt`, `api.surface`, `topology`, etc.) do not appear anywhere in the 25 real session records examined. So the trials were structurally biased toward "does not apply" or "decorative":

- **Cause 1:** Pattern-matching cannot find keywords that do not exist in the input.
- **Cause 2:** Even where patterns matched by accident (e.g., `OPTIONS` → `option` keyword), the sessions produced no structured metrics. C.1's gating model requires conventional evidence (metrics, logs, static-analysis reports); without it, any output is vacuous per C.1's Scenarios 1 and 4.

**Honest statement:** The trials may not give the physics adapter a fair shot. A tool designed to pattern-match on rich engineering language cannot succeed when fed procedural metadata.

---

## Why Deprecate Stands Despite the Confound

**Principle:** A tool that requires different inputs than the system produces is not a deployable tool—it is a research instrument.

Even with this confound in mind, the deprecate verdict holds because:

1. **The corpus IS the real-world data.** The session records in `.mcp-ai-agent-guidelines/session-*.json` are generated by the actual production routing engine. If the system's real telemetry does not include the keywords and metrics the adapter needs, then the adapter cannot operate in production, period. Richer inputs *might* unlock load-bearing mappings, but inputs that rich do not exist in the deployed system today.

2. **The gate already knew this.** Trials A and B both ran `PHYSICS AUDIT (OPT-IN)` and `PHYSICS SCAN (OPT-IN)` gates that returned "0 serial step(s) executed." The gating logic itself (before C.3's decision) already prevented the adapter from executing in a production run. The trials confirmed what the runtime gates already signal: the input shape is not met.

3. **Decorative output is not production-grade.** Even Trial A's QM `review-impact` mapping, which triggered a pattern match, scored decorative: it relabeled standard review practice ("inspect adjacent modules after a change") without adding new evidence or falsifiable predictions. Decorative recommendations waste routing bandwidth and add noise to the skill-selection process.

**Conclusion:** Deprecate as a production tool, not because the concept is flawed, but because the deployed input telemetry does not match the tool's requirements.

## Re-evaluation Trigger

The deprecation is **not permanent**. The adapter may be re-introduced to the routing surface if:

**Condition:** Track A (or another workstream) enriches the session telemetry such that one or both of the following are true:

1. Real session records begin to include domain-specific engineering keywords from the 12 PhysicsConcern patterns (`flak`, `coupling`, `debt`, `api.surface`, `topology`, `shockwave`, etc.) in structured fields (e.g., `context.engineeringConcern`, `request.domain`).

2. Real session records include structured metrics or artifacts: coupling measurements, complexity scores, test failure logs, static-analysis reports, or API-surface counts.

**Next step:** If either condition is met, re-run C.2 trials on a session corpus that includes the new telemetry. If ≥ 2 load-bearing mappings result, invoke Track C.4b to re-introduce the adapter to the routing surface.
